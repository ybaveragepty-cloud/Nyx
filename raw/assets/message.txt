3. My current definition of the 10 core entities
This is how I currently understand them.

You can now adjust or correct them.

1. Farm
My definition
A Farm is the top-level operating entity inside Terra.

It is the parent record that owns:

zones

production units

crops

tasks

alerts

inventory

orders

Example
La Buschagne Aquaponic Farming

Purpose
This is the main container for all farm data.

2. Zone
My definition
A Zone is a major physical or logical area inside a farm.

Examples:

Tunnel Area A

Propagation Zone

Packhouse Zone

Hydroponics Block 1

Purpose
Used to organize the farm into meaningful operational sections.

A zone contains production units.

3. Production Unit
My definition
A Production Unit is the smallest practical grow area that can be planned, assigned, and measured.

Examples:

NFT Line 3

Bed A12

DWC Raft 5

Greenhouse Table 2

Purpose
This is where crop capacity, crop assignment, and rotation planning happen.

This is a critical entity.

4. Crop
My definition
A Crop is the standardized crop type or variety that can be grown.

Examples:

Lettuce

Basil

Spinach

Baby Gem Lettuce

Purpose
Stores the reference information for that crop:

cycle length

expected yield

temperature preference

spacing assumptions

standard growth data

This is the crop template, not the live planted batch.

5. Crop Batch
My definition
A Crop Batch is a live planted instance of a crop in a specific production unit at a specific time.

Example:

Baby Gem batch planted on 2026-03-10 in Tunnel A Line 4

Purpose
Tracks:

planting date

status

expected harvest

actual harvest

yield

issues

batch performance

Crop = template
Crop Batch = actual grow event

6. Task
My definition
A Task is a piece of work assigned to a user or role.

Examples:

transplant bed A12

check EC in tunnel 2

harvest basil line 3

restock packaging

Purpose
Tracks work execution and daily routines.

Tasks can be linked to:

zone

production unit

crop batch

inventory item

alert

7. Alert
My definition
An Alert is a warning, exception, or required attention event created by the system or a workflow.

Examples:

pH out of range

low stock

harvest overdue

task overdue

temperature high

Purpose
Tells the system and users that something needs action.

An alert may create a task.

8. Inventory Item
My definition
An Inventory Item is any tracked input, stock, or consumable used in farm operations.

Examples:

lettuce seed

calcium nitrate

fish feed later

packaging bags

pH buffer

spare pump fittings

Purpose
Tracks quantity, location, reorder threshold, and usage.

9. Customer
My definition
A Customer is a buyer or account that places orders or receives products.

Examples:

restaurant

retailer

distributor

household client

Purpose
Stores contact and relationship information linked to orders.

10. Order
My definition
An Order is a request from a customer for one or more products, with quantity, timing, and fulfillment status.

Examples:

weekly lettuce order

once-off spinach order

standing basil order

Purpose
Tracks commercial demand and links it back to production planning and dispatch.

Yes. That should be included.

What you are describing is the location hierarchy, and it is critical. Without it, the app cannot tell a worker exactly where to go.

Your example:

Farm A → Field A → Block A → Section A → Row 1, 2, 3

is valid and should sit under Site Structure / Production Space.

A good farm app needs two things at the same time:

Production system hierarchy
tells the software what kind of farming method it is

Physical location hierarchy
tells the worker exactly where to go

Best practice

Do not hard-code only “field, block, section, row” for the whole app.

That will break across industries.

Instead, build it as a flexible location tree where each farm type can rename the levels:

open field: Field → Block → Section → Row

hydroponics: Tunnel → Bay → Zone → Channel

aquaculture: Hall → System → Tank

poultry: House → Pen

mushrooms: Room → Rack → Shelf

algae: Reactor Hall → Reactor Line → Reactor Unit

That is the correct approach.

A worker instruction should then read like this:

Go to [Site] > [Area] > [Sub-area] > [Unit] and perform [task].

Example:
Go to Farm A > Field A > Block A > Section A and harvest Rows 1, 2, and 3.

So yes, this belongs in your app, and it is a major part of the design.

The clean way to think about it is:

Site Structure = where things are
Operations = what must be done there

The sentence you give the worker is built from both.
