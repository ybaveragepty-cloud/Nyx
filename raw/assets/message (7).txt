3) Best install method for you
Because you already have Ubuntu + Docker + Coolify, use the Docker Compose OpenClaw gateway setup. The official Docker guide says this is the right path if you want a containerized gateway, and the quickest recommended flow is ./docker-setup.sh, which builds or pulls the image, runs onboarding, starts the gateway, and writes the gateway token to .env. 

4) Step-by-step installation on Ubuntu VPS
Step 1 — Prepare a dedicated OpenClaw directory
On the VPS:

cd /opt
sudo git clone https://github.com/openclaw/openclaw.git
cd openclaw
Use a dedicated directory. Do not mix this with your future YBAFlow repo.

Step 2 — Use the prebuilt image to reduce build pain
OpenClaw’s Docker docs say you can skip local image build time by setting OPENCLAW_IMAGE="ghcr.io/openclaw/openclaw:latest" before running setup. 

export OPENCLAW_IMAGE="ghcr.io/openclaw/openclaw:latest"
Step 3 — Mount only what the agent needs
The Docker guide supports OPENCLAW_EXTRA_MOUNTS to bind host directories into the gateway and CLI containers. 

For your use case, create one workspace root for build work:

sudo mkdir -p /srv/ai-workspaces
sudo mkdir -p /srv/ai-workspaces/ybaflow-core
sudo mkdir -p /srv/ai-workspaces/ybaterra
sudo chown -R $USER:$USER /srv/ai-workspaces
Then mount only that root:

export OPENCLAW_EXTRA_MOUNTS="/srv/ai-workspaces:/home/node/projects:rw"
Do not mount /, /root, or your whole /opt.

Step 4 — Run the OpenClaw setup script
Official quick start for Docker:

./docker-setup.sh
That script:

builds or pulls the image

runs onboarding

starts the gateway

generates a gateway token into .env 

Step 5 — Start or verify the gateway
If you ever need the manual flow, the docs show:

docker build -t openclaw:local -f Dockerfile .
docker compose run --rm openclaw-cli onboard
docker compose up -d openclaw-gateway
And to verify the gateway:

docker compose ps
docker compose logs -f openclaw-gateway
The docs also note a built-in healthcheck and a health command for the gateway. 

Step 6 — Open the Control UI
OpenClaw’s getting started docs say the fastest test is the Control UI, and openclaw dashboard opens it. If Docker pairing fails, the Docker docs show how to fetch and approve a device request. 

Docker flow:

docker compose run --rm openclaw-cli dashboard --no-open
docker compose run --rm openclaw-cli devices list
docker compose run --rm openclaw-cli devices approve <requestId>
5) Hardening and safe operating mode
This matters. A lot.

The Docker docs warn that public-host deployments need security hardening, and OpenClaw’s sandbox docs explain that agents.defaults.sandbox.mode controls whether sessions run in Docker-isolated sandboxes. 

For your build setup, use:

main admin session: host access allowed

all non-main sessions: sandboxed

public channels: off for now unless you really need them

Recommended mode:

{
  "agents": {
    "defaults": {
      "sandbox": {
        "mode": "non-main",
        "scope": "agent"
      }
    }
  }
}
That matches the documented config direction for sandbox mode and scope. 

6) Multi-model API infrastructure for your build system
This is the right place to be disciplined.

OpenClaw supports many providers, including OpenAI, Anthropic, Gemini, OpenRouter, Groq, Mistral, Ollama, Vercel AI Gateway, and others. It also supports:

provider/model naming

agents.defaults.models as an allowlist

primary + fallback model selection

API key rotation for selected providers 

Recommended provider stack for you
Use three tiers:

Tier A — primary high-quality build model

Anthropic API or OpenAI API

Tier B — fallback

Gemini API or OpenRouter

Tier C — cheap utility / classification / summarization

Groq or Ollama/local later

Recommended model policy
Architecture / design / refactor / reasoning → strongest model

Code generation / migrations / test writing → strong code-capable model

Cheap repetitive tasks → lower-cost fallback model

The model selection docs explicitly support primary and fallback model chains. 

Example environment variables
export OPENAI_API_KEY="..."
export ANTHROPIC_API_KEY="..."
export GEMINI_API_KEY="..."
export OPENROUTER_API_KEY="..."
Example policy config
Use a config like this in ~/.openclaw/openclaw.json:

{
  "agents": {
    "defaults": {
      "workspace": "/home/node/projects",
      "model": {
        "primary": "anthropic/claude-opus-4-6",
        "fallbacks": [
          "openai/gpt-5.1-codex",
          "google/gemini-3-pro-preview",
          "openrouter/anthropic/claude-sonnet-4-5"
        ]
      },
      "models": {
        "anthropic/claude-opus-4-6": {},
        "openai/gpt-5.1-codex": {},
        "google/gemini-3-pro-preview": {},
        "openrouter/anthropic/claude-sonnet-4-5": {}
      },
      "sandbox": {
        "mode": "non-main",
        "scope": "agent"
      }
    }
  }
}
This follows the documented provider/model format, allowlist behavior, and primary/fallback approach.
