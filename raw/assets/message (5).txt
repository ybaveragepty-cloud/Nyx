OpenClaw Agent Task Plan for Terra
Goal
Build Terra as a stable, modular, multi-tenant farm operations platform with:

customizable dashboards

customizable workflows

strict role/permission control

simple worker UI

crop and livestock domain depth

auditability

future AI/automation readiness

Delivery rule for all agents
Every task must produce:

code

tests

migration or schema changes if relevant

short implementation note

update to task status

No agent may skip to UI polish before the relevant data model, API contract, permissions, and tests exist.

Phase 1 — Foundation and build rules
1–10

Set up apps, packages, shared types, UI library, backend services, workflow engine area, and docs folders.

2. Define coding standards and contribution rules

Document frontend, backend, database, workflow engine, auth, notifications, and AI integration boundaries.

4. Create environment configuration system
Centralize env validation for dev, staging, and production.

5. Create logging and error handling standards

6. Set up automated testing pipeline
Configure unit, integration, API, and end-to-end test execution.

7. Set up CI pipeline
Run install, lint, typecheck, tests, migration checks, and build on each commit.

8. Set up CD/release pipeline skeleton
Create deployment workflow for staging and production.

9. Create shared design tokens and UI primitives
Set typography, spacing, colors, states, form components, tables, cards, dialogs, badges.

10. Create system-wide feature flag framework
Allow phased rollout of incomplete modules without breaking the platform.

Phase 2 — Core tenancy and identity
11–20
11. Design the multi-tenant data model

12. Create organization onboarding model

13. Build authentication service integration

14. Create user profile model
Store user identity, organization membership, role assignments, status, and preferences.

15. Create role model
Support admin, manager, supervisor, worker, viewer, and future custom roles.

16. Create permission matrix engine
Support action-level permissions, not just page-level access.

17. Build organization membership management APIs
Invite users, assign roles, disable accounts, transfer org ownership rules.

18. Build admin-only permission assignment UI

20. Create audit log for auth and role changes

Phase 3 — Master entity model
21–30

Phase 4 — Workflow engine and configurability
31–40

The full master model is split into 12 domains:

Tenant and Identity

2. Core Design Rules
Rule 1
Every record belongs to an organization.

Rule 2
Most operational records should also link to a site and often a more specific location node.

Rule 3
Anything that is produced, moved, treated, harvested, packed, or sold should have a batch / lot / cohort identity.

Rule 4
Every action should be traceable back to:

who did it

when they did it

where it happened

what it affected

Rule 5
No workflow record should exist without linking to a real business object:

crop cycle

fish cohort

inventory item

cleaning event

maintenance event

order

dispatch

etc.

Rule 6
Configuration must be tenant-aware:
each organization can customize screens, workflows, permissions, KPIs, and naming.

18. High-Risk Areas to Control Early
These are the places where agents usually make a mess.

1. Location model
Do not hardcode greenhouse-only logic.
Use a generic location tree.

2. Batch model
Do not create separate random batch systems for crops, fish, and packhouse.
Use one universal batch/lot concept with typed behavior.

3. Workflow model
Do not attach workflow directly to pages.
Attach workflow to entity types and real operations.

4. Dashboard model
Do not hardcode one dashboard per role forever.
Use widget + layout + permission model.

Do not use only frontend hiding.

Do not let every customer change schema directly.
Use custom fields, config, and versioned workflow definitions.

These tables must exist before most others:

organization

app_user

organization_membership

site

location_node

6.2 Universal FK pattern
Operational tables should link to:

organization_id

site_id when physical

location_node_id when local

responsible membership where human execution matters

6.3 Polymorphic references
Allowed in v1 only where necessary:

source_type + source_reference_id

linked_entity_type + linked_entity_id

reference_type + reference_id

task_source_type + task_source_id

These must be controlled in service logic, not left loose everywhere.

6.4 Self-referencing tables
Allowed:

location_node.parent_location_node_id

biological_batch.parent_batch_id

asset_category.parent_category_id

7. Index strategy
High priority indexes
Must exist early:

all FK columns

task_instance(status, assigned_to_membership_id)

measurement_record(measurement_type_code, measured_at)

stock_transaction(reference_type, reference_id)

audit_log(entity_type, entity_id, timestamp)

hardcoding greenhouse-only structures

tenant custom config

Change control
Any new core table after approval of v1 must include:

purpose

owning module

why existing tables are insufficient

FKs

migration placement

effect on audit and permissions
