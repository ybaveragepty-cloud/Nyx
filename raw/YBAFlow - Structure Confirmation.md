## 4) Step-by-step installation on Ubuntu VPS

### Step 1 — Prepare a dedicated OpenClaw directory

On the VPS:

Bash

cd /opt  
sudo git clone https://github.com/openclaw/openclaw.git  
cd openclaw

Use a **dedicated directory**. Do not mix this with your future YBAFlow repo.

### Step 2 — Use the prebuilt image to reduce build pain

OpenClaw’s Docker docs say you can skip local image build time by setting `OPENCLAW_IMAGE="ghcr.io/openclaw/openclaw:latest"` before running setup. [GitHub](https://github.com/openclaw/openclaw/blob/main/docs/install/docker.md)

Bash

export OPENCLAW\_IMAGE\="ghcr.io/openclaw/openclaw:latest"

### Step 3 — Mount only what the agent needs

The Docker guide supports `OPENCLAW_EXTRA_MOUNTS` to bind host directories into the gateway and CLI containers. [GitHub](https://github.com/openclaw/openclaw/blob/main/docs/install/docker.md)

For your use case, create **one workspace root** for build work:

Bash

sudo mkdir \-p /srv/ai-workspaces  
sudo mkdir \-p /srv/ai-workspaces/ybaflow-core  
sudo mkdir \-p /srv/ai-workspaces/ybaterra  
sudo chown \-R $USER:$USER /srv/ai-workspaces

Then mount only that root:

Bash

export OPENCLAW\_EXTRA\_MOUNTS\="/srv/ai-workspaces:/home/node/projects:rw"

Do **not** mount `/`, `/root`, or your whole `/opt`.

### Step 4 — Run the OpenClaw setup script

Official quick start for Docker:

Bash

./docker-setup.sh

That script:

- builds or pulls the image
- runs onboarding
- starts the gateway
- generates a gateway token into `.env` [GitHub+2GitHub+2](https://github.com/openclaw/openclaw/blob/main/docs/install/docker.md)

### Step 5 — Start or verify the gateway

If you ever need the manual flow, the docs show:

Bash

docker build \-t openclaw:local \-f Dockerfile .  
docker compose run \--rm openclaw-cli onboard  
docker compose up \-d openclaw-gateway

And to verify the gateway:

Bash

docker compose ps  
docker compose logs \-f openclaw-gateway

The docs also note a built-in healthcheck and a health command for the gateway. [GitHub](https://github.com/openclaw/openclaw/blob/main/docs/install/docker.md)

### Step 6 — Open the Control UI

OpenClaw’s getting started docs say the fastest test is the Control UI, and `openclaw dashboard` opens it. If Docker pairing fails, the Docker docs show how to fetch and approve a device request. [OpenClaw+1](https://docs.openclaw.ai/start/getting-started?utm_source=chatgpt.com)

Docker flow:

Bash

docker compose run \--rm openclaw-cli dashboard \--no-open  
docker compose run \--rm openclaw-cli devices list  
docker compose run \--rm openclaw-cli devices approve <requestId>

## 5) Hardening and safe operating mode

This matters. A lot.

The Docker docs warn that public-host deployments need security hardening, and OpenClaw’s sandbox docs explain that `agents.defaults.sandbox.mode` controls whether sessions run in Docker-isolated sandboxes. [GitHub+2OpenClaw+2](https://github.com/openclaw/openclaw/blob/main/docs/install/docker.md)

For your build setup, use:

- **main admin session**: host access allowed
- **all non-main sessions**: sandboxed
- **public channels**: off for now unless you really need them

Recommended mode:

JSON

{  
  "agents": {  
    "defaults": {  
      "sandbox": {  
        "mode": "non-main",  
        "scope": "agent"  
      }  
    }  
  }  
}

That matches the documented config direction for sandbox mode and scope. [OpenClaw+1](https://docs.openclaw.ai/gateway/sandboxing?utm_source=chatgpt.com)

## 6) Multi-model API infrastructure for your build system

This is the right place to be disciplined.

OpenClaw supports many providers, including OpenAI, Anthropic, Gemini, OpenRouter, Groq, Mistral, Ollama, Vercel AI Gateway, and others. It also supports:

- `provider/model` naming
- `agents.defaults.models` as an allowlist
- primary + fallback model selection
- API key rotation for selected providers [OpenClaw+2OpenClaw+2](https://docs.openclaw.ai/concepts/model-providers)

### Recommended provider stack for you

Use **three tiers**:

**Tier A — primary high-quality build model**

- Anthropic API or OpenAI API

**Tier B — fallback**

- Gemini API or OpenRouter

**Tier C — cheap utility / classification / summarization**

- Groq or Ollama/local later

### Recommended model policy

- **Architecture / design / refactor / reasoning** → strongest model
- **Code generation / migrations / test writing** → strong code-capable model
- **Cheap repetitive tasks** → lower-cost fallback model

The model selection docs explicitly support primary and fallback model chains. [OpenClaw](https://docs.openclaw.ai/concepts/models?utm_source=chatgpt.com)

- allowed repos
- allowed commands
- banned directories
- banned deployment targets
- migration rules
- test requirements before commit

This is where you make OpenClaw useful instead of dangerous.
